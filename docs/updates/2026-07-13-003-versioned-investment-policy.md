## 为什么这项能力现在必须做

生产数据表明，现有基金 Agent Run 全部停留在 `setup_required` 或 `research_only`。原因不是缺少更多行情指标，而是系统没有一份经过用户确认、可追溯且可固定引用的投资约束。

如果没有正式投资政策书，Agent 无法可靠回答：

- 这笔资金可以承担多大回撤。
- 资金多久以后需要使用。
- 每月实际可投入多少，不影响应急资金。
- 单只基金、权益资产和单行业最多占多少。
- 是否允许港股、美股或全球基金及其汇率风险。
- 用户经验是否与进取型风险配置匹配。
- 某次 Agent Run 当时使用的是哪一版约束。

继续增加预测模型只会让输出看起来更复杂，不能让建议变得更适合用户。IPS 是从公共基金研究走向个人决策的必要门禁，但它不保证收益。

## 为什么本轮没有先做登录

身份与数据隔离仍然是工业级产品的最高优先级之一，但当前生产站点只有 IP 和 HTTP。此时直接上线密码登录会让凭据在网络中缺少 TLS 保护，不符合生产安全要求。

正确顺序是：

1. 本轮先把单用户约束升级为版本化、不可变、可审计事实。
2. 用户准备域名并启用 HTTPS。
3. 再上线身份、会话、用户隔离、密码/MFA 和数据迁移。

本轮没有把“登录按钮”当作安全能力演示。

## 旧实现的问题

旧 `investment_profiles` 表每个用户只有一行：

- 每次保存覆盖上一份值，无法复核历史。
- 保存即生效，没有草稿、校验和显式确认。
- 没有载荷哈希和审计链。
- 没有乐观并发，两个页面可互相覆盖。
- Agent Run 不记录使用的档案版本。
- 仅包含风险、期限、月度预算和单品上限，缺少流动性、回撤、经验和组合边界。

因此，旧表不再作为政策事实来源，只保留为兼容投影。

## 两阶段生命周期

```text
用户填写问卷
  -> 确定性标准化
  -> 适当性校验
  -> 生成不可变草稿与 SHA-256
  -> 用户查看错误、警告、版本号和哈希
  -> 用户明确勾选固定版本同意条款
  -> 校验预期生效版本（乐观并发）
  -> 原版本 superseded
  -> 草稿 active
  -> 更新兼容投影
  -> 追加哈希审计事件
```

草稿创建永远不会让 Agent 开始使用该政策。只有通过校验、哈希一致、同意条款一致且并发基线未变化的草稿才能激活。

## 政策字段

| 字段 | 作用 |
|---|---|
| `risk` | stable、balanced、aggressive |
| `horizon` | short、mid_long、long |
| `experience_level` | beginner、intermediate、experienced |
| `primary_objective` | 保本优先、平衡增长、长期增长 |
| `monthly_budget` | 用户明确确认的月度新增预算，可为 0 |
| `max_single_ratio` | 单只基金上限 |
| `max_equity_ratio` | 权益资产总上限 |
| `max_industry_ratio` | 单行业上限 |
| `max_drawdown_pct` | 用户确认的最大可承受回撤 |
| `liquidity_reserve_months` | 不参与投资的流动性储备月份 |
| `allowed_fund_markets` | 内地、香港、美国、全球 |
| `accept_fx_risk` | 是否接受跨境汇率和净值时差风险 |
| `emergency_fund_confirmed` | 投资预算不占用应急资金的确认 |
| `review_cycle_months` | 6 或 12 个月后强制复核 |

所有字段由用户确认。系统不会根据年龄、持仓或历史行为推断并写入 IPS。

## 确定性适当性规则

通用门禁：

- 所有数值必须是有限数值并在固定范围内。
- 至少选择一个基金市场。
- 跨境市场必须明确接受汇率和净值确认风险。
- 流动性储备至少 3 个月。
- 必须确认预算不占用应急资金。
- 单行业上限不能高于权益资产总上限。
- 复核周期只能是 6 或 12 个月。

风险等级上限：

| 风险等级 | 单品上限 | 权益上限 | 最大回撤 |
|---|---:|---:|---:|
| stable | 30% | 40% | 15% |
| balanced | 45% | 75% | 30% |
| aggressive | 60% | 100% | 50% |

组合冲突：

- `beginner + aggressive` 不允许激活。
- `short + aggressive` 不允许激活。
- 仅内地市场却确认汇率风险会产生警告，但不是错误。
- 全球市场会提示多币种、额度和交易日历风险。

这些规则是版本化确定性门禁，不由大模型自由解释。

## 数据模型与不可变性

新增 `investment_profile_versions`：

- 每个用户单调递增 `version_no`。
- 状态只允许 `draft -> active -> superseded`。
- 每个用户最多一个 `active` 版本。
- 保存规范化 `payload_json`、`payload_sha256` 和校验结果。
- 保存问卷版本、同意条款版本和条款文本哈希。
- 保存创建、激活、复核到期和取代时间。

数据库触发器阻止：

- 修改版本载荷、载荷哈希、校验结果、版本号和创建时间。
- 非法状态回退或跳转。
- 版本生效后修改同意条款、激活时间或复核日期。

`investment_profiles` 只在版本激活事务成功后更新，用于兼容旧组合模块。

## 审计链与联合完整性

新增 `investment_profile_audit_events`，按用户维护连续 SHA-256 链：

- `profile.draft_created`
- `profile.version_activated`
- `profile.version_superseded`
- `profile.legacy_migrated`

联合校验不仅验证事件链，还验证：

- 每个版本的载荷哈希。
- 每个版本都有创建或迁移事件。
- active/superseded 版本都有激活或迁移事件。
- 最多一个 active 版本。

任一校验失败时，`configured=false`，Agent 停止使用该 IPS。

## 并发与幂等

- 相同载荷哈希的现有草稿会直接复用。
- 重复激活已生效且哈希一致的版本返回幂等成功，不增加事件。
- 激活请求必须携带页面读取到的 `expected_active_version_id`。
- 如果另一页面先激活了新版本，旧页面收到 HTTP 409，必须重新核对，不能覆盖。
- 草稿哈希必须与激活请求中的哈希一致。
- 激活在 SQLite `BEGIN IMMEDIATE` 事务中完成。
- 存储连接启用 WAL、外键、30 秒 busy timeout。

## 旧数据迁移

如果旧 `investment_profiles` 存在记录：

- 保留原风险、期限、预算、单品上限和市场权限。
- 不猜测投资经验、流动性、权益上限、行业上限或回撤承受能力。
- 创建 `legacy_incomplete` 版本及迁移审计事件。
- `configured=false`，要求用户重新确认完整问卷。

如果旧表没有记录，不创建任何版本或审计事件。

## Agent 固定版本引用

`agent_runs` 新增 `profile_version_id`：

1. 创建 Run 时，服务只在当前 IPS 已激活、未过期且联合完整性通过时绑定版本 ID。
2. Run 输入和数据库列同时保存该 ID。
3. `portfolio.context.get` 只读取该精确版本，不回退到后来激活的新版本。
4. Portfolio Context Evidence 保存版本号、载荷哈希、激活时间、复核时间和同意版本。
5. 即使该版本后来变成 `superseded`，历史 Run 仍能复核当时约束。
6. 重跑使用重跑时当前有效 IPS，并在 Run 对比中明确显示政策版本变化。

这避免历史建议被当前设置反向改写。

## 决策策略 1.1.0

个性化基金决策新增 `drawdown_capacity` 硬门禁：

- 读取真实基金历史最大回撤。
- 与当前 Run 固定 IPS 中的 `max_drawdown_pct` 比较。
- 基金历史最大回撤超过用户承受上限时，决策为 `do_not_add`。
- 不生成新增金额。

权益总上限和行业上限已经进入 IPS 与 Evidence，但当前不假装实时执行。原因是组合基金穿透数据还不能完整证明所有底层权益和行业暴露；这两个约束将在组合穿透快照版本化后接入硬门禁。

## API

- `GET /api/investment-profile`：读取当前生效状态；过期或完整性失败时 `configured=false`。
- `PUT /api/investment-profile`：兼容入口，现只创建草稿，不再直接生效。
- `POST /api/investment-profile/drafts`：校验并创建不可变草稿。
- `GET /api/investment-profile/versions`：读取脱敏版本元数据，不返回历史问卷载荷。
- `POST /api/investment-profile/versions/{version_id}/activate`：哈希、同意和并发校验后激活。
- `GET /api/investment-profile/audit`：读取审计事件和联合完整性结果。

## 前端

决策中心的投资约束区升级为 IPS 工作区：

- 展示当前版本、复核日期、载荷完整性和审计状态。
- 填写风险、期限、经验、目标、预算、集中度、回撤和流动性约束。
- 第一步只生成草稿。
- 展示字段级错误、警告、版本号和完整载荷哈希。
- 校验通过后显示固定版本同意条款。
- 用户勾选后才允许激活。
- 展示最近版本的状态、时间和哈希前缀。
- 桌面、平板和手机均有固定响应式布局。

## 测试证据

覆盖：

- 规范化载荷哈希稳定。
- 跨境市场必须有汇率风险确认。
- 风险等级、经验、期限和流动性冲突。
- 草稿不影响当前 Agent。
- 无确认不能激活。
- 无效草稿不能激活。
- 相同草稿和重复激活幂等。
- 并发页面旧版本激活冲突。
- 数据库载荷与激活元数据不可修改。
- 审计篡改可检测且会使当前 IPS 失效。
- 旧档案迁移后必须重新确认。
- 已被取代的版本仍可被历史 Agent 精确读取。
- Agent Run 同时在输入和数据库列固定版本 ID。
- 重跑对比显示 IPS 版本变化。
- 基金历史回撤超过 IPS 上限时禁止新增。
- 原 Agent、基金、组合、调度和 API 路由回归。

本地验收：

- 后端全量测试：113 项通过。
- 前端 `npm run build`：通过。
- 真实本地数据库当前没有旧 IPS：版本 0、审计事件 0、`configured=false`。
- 历史 Run 成功新增 `profile_version_id` 列，原状态和 Evidence 未改变。
- 没有代替用户创建或激活投资政策。

## 已知限制与下一步

- 当前仍是单用户迁移存储，尚未完成身份和租户隔离。
- 在域名与 HTTPS 就绪前，不上线密码登录。
- SQLite 适合当前单机；迁移 PostgreSQL 后应增加外键、RLS 和正式迁移工具。
- IPS 只定义风险边界，不代表任何策略能够盈利。
- 下一步候选是“组合穿透约束执行”：用版本化真实持仓穿透快照执行权益和行业上限，而不是只保存字段。

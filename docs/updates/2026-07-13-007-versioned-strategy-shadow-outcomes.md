# 版本绑定的策略 Shadow Outcome 账本

## 1. 更新信息

- 更新编号：`2026-07-13-007`
- 核心能力：按策略精确版本自动入组、冻结信号、等待真实确认净值、生成不可变 Outcome Evidence
- 入组策略：`strategy_shadow_enrollment@1.0.0`
- 结果评估器：`fund_strategy_shadow_outcome@1.0.0`
- 工具：`fund.strategy_shadow_outcome.get@1.0.0`
- 结果协议：`strategy_shadow_outcome.v1`
- 汇总协议：`strategy_shadow_report@1.0.0`
- 数据原则：真实确认净值、来源原生同类比较、禁止近似日期、禁止模拟收益
- 发布影响：无。达到统计披露门槛也不会自动把策略升级为 Canary 或 Active

## 2. 本轮为什么必须做

上一轮已经建立策略注册表、版本清单、审计链和发布门禁，但 Shadow 策略被正确禁止进入个人金额决策后，也无法进入原有的个人行动 Outcome 流程。若不增加独立的策略观察账本，会形成闭环断点：

1. 未发布策略不能影响真实投入金额，这是正确的安全约束。
2. 原有 Outcome 只跟踪已经发布且产生个人方向动作的决策。
3. Shadow 策略因此永远没有系统化的样本外结果。
4. `minimum_shadow_outcomes` 发布检查永远不可能获得真实证据。

这意味着继续增加选股、基金推荐或更激进的金额建议，只会放大未经验证策略的风险。当前最有价值的工程工作不是制造一个更好看的“预测准确率”，而是建立从信号产生、等待真实结果、证据固化、完整性验证到统计披露的闭环。

该能力不保证盈利。它的直接价值是阻止无证据策略进入资金决策，并为后续淘汰无效策略、识别稳定超额能力和控制实盘风险提供可审计数据。

## 3. 被否决的方案

### 3.1 复用个人行动 Outcome

个人行动 Outcome 表达“系统给某个用户的金额动作后来如何”，策略 Shadow Outcome 表达“某个策略精确版本当时产生的方向信号后来如何”。二者混用会污染语义，还会把用户是否执行、何时执行和执行成本错误归因给策略。

### 3.2 由管理员手工挑选样本

手工挑选容易只保留表现好的信号，造成选择偏差。当前实现只允许终态 Run 按完成时间顺序自动入组；公网没有创建、补录、提前结算或修改基线的接口。

### 3.3 立即展示胜率和平均收益

少量、重叠且集中于同一市场环境的样本不能证明策略有效。当前至少需要 30 个完整同类基准样本、覆盖 10 只不同基金、全量扫描完成且无完整性错误，才允许返回汇总指标。

### 3.4 用自然日或最近交易日代替观察窗口

QDII、港股、美股、A 股基金的确认净值日历不同。使用“约三个月后的最近净值”会让不同基金的观察长度不一致。本轮使用冻结基线之后第 N 个真实确认净值作为唯一目标。

### 3.5 用合成指数或静态常数补齐基准

当来源原生同类比较不可用时，系统保留方向结果，但标为非 release-grade；不使用模拟指数、静态收益率或跨来源拼接数据补齐发布样本。

## 4. 适用范围与非目标

本轮支持：

- `fund_deep_research.v4` 终态 Run。
- 已注册且处于 `shadow`、`canary` 或 `active` 的精确策略版本。
- `positive` 和 `negative` 两种明确方向。
- 策略主窗口及其确认净值观测数。
- 基金单位净值方向结果和来源原生同类平均相对结果。
- 自动入组、后台观察、失败重试、证据固化、审计回放和只读查询。

本轮不做：

- 不自动交易，不连接券商或基金申赎。
- 不推断用户实际成交、手续费、税费、滑点或持有收益。
- 不把单次结果称为策略胜率或盈利证明。
- 不把达到披露门槛等同于策略发布审批。
- 不为缺失基准、缺失净值或日期不匹配制造兜底数据。
- 不处理同一基金不同份额之间的费用归一化。

## 5. 端到端流程

```text
终态 fund_deep_research.v4 Run
        |
        v
校验策略版本、Manifest 哈希、治理 Evidence、信号 Evidence、Run 审计链
        |
        v
冻结 Signal Snapshot + SHA-256
        |
        v
按 strategy_version + fund + horizon 检查非重叠窗口
        |
        +---- 窗口重叠 ----> excluded（保留原因和阻塞样本 ID）
        |
        v
scheduled -> leased -> provider evaluation
                     |
                     +---- 未到第 N 个确认净值 -> pending/retry_wait
                     +---- 基线被改写或证据损坏 -> blocked
                     +---- 到达精确目标 -> observed + immutable Evidence
                                                     |
                                                     v
                                  完整性扫描与统计披露门槛
```

## 6. 不可变入组快照

每个入组样本冻结以下内容：

| 分类 | 冻结字段 |
|---|---|
| 来源 Run | Run ID、完成时间、结果协议版本、入组前工作流审计链头 |
| 策略身份 | 策略 ID、语义化版本、Manifest SHA-256、信号时策略状态 |
| 治理状态 | 是否允许计算、是否允许进入金额决策、发布检查状态 |
| Evidence | 治理 Evidence ID、信号 Evidence ID、Run Evidence ID 集合 |
| 基金基线 | 基金代码、名称、确认净值日期、单位净值 |
| 信号 | 方向、研究动作、置信级别、条件状态 |
| 观察窗口 | 主窗口名称、后续确认净值数量 |
| 审计 | 入组策略版本、快照 SHA-256 |

快照使用键排序、无多余空白的规范 JSON 计算 SHA-256。Worker 执行前会重新计算；任何字段被直接修改都会阻断观察，不会继续请求数据源。

审计链头特意冻结在第一个 `strategy.shadow.*` 事件之前，避免后续入组和观察事件反过来改变原始信号快照。

## 7. 样本选择与非重叠规则

自动回填按 Run 完成时间正序扫描，最多处理 1000 个尚未评估的历史 Run。无效 Run 不会阻塞游标继续扫描后续有效 Run。

非重叠单元为：

```text
strategy_id + strategy_version + fund_code + horizon
```

同一个单元在已有未完成窗口时，新 Run 仍会持久化，但状态为 `excluded`。只有旧窗口已经到达真实观察日，且新 Run 的基线晚于旧观察窗口，才允许生成下一个有效样本。

该规则的目标是降低高度重叠前瞻收益带来的虚假样本量。它不声称跨基金样本完全独立，市场环境相关性仍作为已知限制保留。

## 8. 精确结果评估

### 8.1 目标日

目标是冻结基线之后第 N 个有效确认净值：

```text
target = Nth confirmed NAV strictly after frozen baseline
```

例如窗口为 63，则只使用第 63 个后续确认净值。即使数据源已经返回第 64、65 个净值，也不会改用更晚日期。

### 8.2 基线校验

数据源必须仍能返回冻结日期的单位净值。允许的数值误差为 `max(1e-6, |baseline| * 1ppm)`。超过误差视为来源历史发生修订，状态转为 `blocked`，系统不静默改写原始信号。

### 8.3 方向评分

- `positive`：窗口单位净值收益大于 0 才算方向正确。
- `negative`：窗口单位净值收益小于 0 才算方向正确。
- 方向恰好为 0 不算命中。
- `mixed`、缺失方向或非方向研究结果不能入组。

为了可比较，负向信号使用符号调整收益：负向且实际下跌时，`signed_unit_nav_return_pct` 为正。

### 8.4 同类比较

同类比较必须来自数据提供方原生同类序列，基金和同类序列必须同时包含冻结基线日和观察目标日。任何一端日期缺失都不会使用最近日期代替。

同类比较完整时样本才标记为 `release_grade=true`。同类数据不可用时仍可保存基金方向观察，但不能进入策略发布汇总。

## 9. 状态机、租约与重试

主要状态：

| 状态 | 含义 |
|---|---|
| `scheduled` | 已入组，等待预计窗口日期 |
| `leased` | 某个 Worker 已获得有限时租约 |
| `retry_wait` | 数据暂不可用，按退避时间重试 |
| `observed` | 已到达精确目标并写入 Outcome Evidence |
| `excluded` | 与已有窗口重叠，不参与观察 |
| `blocked` | 非重试错误、证据损坏或连续失败达到上限 |

Worker 通过数据库原子更新领取租约，避免多实例重复处理同一记录。租约过期后可以被其他实例接管。每次执行记录开始、等待、完成或失败审计事件。

可重试错误使用退避时间；连续 8 次失败后进入 `blocked`。非重试错误立即阻断。状态表被直接改写时，审计事件回放与当前状态不一致，读取和汇总都会报告完整性失败。

## 10. Evidence 与审计链

观察完成后生成独立的 `strategy_shadow_outcome` Evidence，包含：

- 精确策略版本和 Manifest 哈希。
- 冻结信号快照哈希。
- 基线和第 N 个确认净值目标。
- 单位净值收益、方向命中结果。
- 同类比较状态、相对收益和 release-grade 状态。
- 数据来源、来源日期和质量信息。
- 评估器版本、限制和方法声明。

Evidence 是追加式记录，不覆盖原信号 Evidence。入组状态迁移同时写入 Run 审计链：

- `strategy.shadow.enrolled`
- `strategy.shadow.observation.started`
- `strategy.shadow.pending`
- `strategy.shadow.completed`
- `strategy.shadow.failed`

查询单个 Run 时，服务重新验证快照哈希、策略 Manifest、来源 Evidence、原工作流审计链和 Shadow 状态回放。失败样本不会静默进入汇总。

## 11. 统计披露门槛

以下条件必须同时满足：

1. 至少 30 个 `release_grade` 观察结果。
2. 至少覆盖 10 只不同基金。
3. 报告扫描完整，没有因 2000 条读取上限被截断。
4. 完整性失败数量为 0。

门槛未满足时，API 的 `metrics` 必须为 `null`，前端只显示样本进度，不显示胜率、平均收益或排行榜。

达到门槛后可披露：

- 方向命中率和 Wilson 95% 区间。
- 符号调整后的平均单位净值收益。
- 完整同类比较样本数。
- 同类超额方向命中率和 Wilson 95% 区间。

这些指标仍不包含用户交易成本、税费、汇率归因和实际执行，也不能自动改变策略发布状态。独立 Reviewer 仍需完成样本独立性、市场分层、成本模型和风险检查。

## 12. API 与前端

新增只读 API：

```text
GET /api/v1/agent/strategies/{strategy_id}/{strategy_version}/shadow-outcomes
GET /api/v1/agent/runs/{run_id}/strategy-shadow-outcome
```

公网没有手工入组、修改、删除、强制执行或提前结算 API。Run 查询响应会移除租户 ID、用户 ID 和原始错误消息，只暴露稳定错误码。

Agent 页面新增只读 Shadow Outcome 面板，展示：

- 是否已入组及当前状态。
- 冻结基线、主窗口和确认净值数量。
- 已观察和完整同类基准样本数。
- 是否达到统计披露门槛。
- 快照与审计是否验证通过。
- 已完成时的 Outcome Evidence 入口。

桌面和 375px 移动端均验证无横向溢出，控制台无 warning 或 error。

## 14. 安全、隐私与运行边界

- 数据库仍为单机 SQLite，适合当前单服务器阶段；多副本写入前应迁移 PostgreSQL。
- Worker 租约提供进程级容错，不替代跨区域任务队列和分布式锁。
- 启动回填单次最多扫描 1000 个 Run；报告最多扫描 2000 条，截断时拒绝披露汇总。
- 当前只评估基金策略主窗口，不评估股票、组合再平衡和多窗口联合决策。
- 数据源许可、频率限制和历史修订需要独立运维监控。
- 港股、美股和 QDII 暴露已可通过基金真实确认净值进入观察，但尚未拆分汇率、交易时区和底层资产贡献。
- 同类平均是数据源提供的比较序列，不是基金合同约定业绩基准。
- 本轮不构成收益承诺或投资建议；策略只有在证据充分、风险检查通过且人工审批后，才可能进入金额决策。

## 15. 回滚与故障处置

回滚顺序：

1. 停止 `StrategyShadowOutcomeWorker`，阻止新任务领取租约。
2. 将应用回滚到上一提交，保留新增表和 Evidence，不删除审计记录。
3. 若数据库迁移异常，使用部署前 SQLite 在线备份恢复。
4. 恢复后运行 `PRAGMA integrity_check`，校验 Run Evidence 和审计链。
5. 重新启用 Worker 前，检查所有 `leased` 记录是否已过期并可安全接管。

回滚不得删除表现不佳的观察结果，也不得重写冻结基线。历史负面证据属于策略治理记录的一部分。

## 16. 下一阶段准入条件

在继续扩大收益决策前，优先完成：

1. 将 Shadow 样本按 A 股、港股、美国市场、QDII、债券和商品暴露分层。
2. 引入基金费用、申赎规则、汇率和用户执行时点的独立成本模型。
3. 增加数据源可用性、净值修订和 Worker 积压的生产监控告警。
4. 建立 PostgreSQL、身份认证、RBAC 和管理端审批流。
5. 为达到披露门槛的策略执行独立样本、市场状态和回撤审查，不直接自动发布。

只有这些基础能力形成闭环后，Agent 才能逐步从“给出研究意见”演进为“在明确风险预算内提供可验证的资金决策支持”。

# 2026-07-15 更新 008：多基金批次统一资金分配

## 1. 目标与边界

本轮解决批量研究中的一个实质性资金风险：同一笔计划投入不能在每只基金的子 Run 中重复出现。

新的职责划分是：

1. 单基金 Run 只判断该基金是否适合进入本批资金复核，并计算可验证的最大候选容量。
2. Batch 在全部子 Run 终态后，只对一笔用户明确确认的总预算做一次联合分配。
3. 分配金额由确定性规则计算，大模型只能做 Evidence 解释，不能修改金额或授权交易。
4. 真实数据、组合版本或联合暴露参数不完整时停止分配，不使用均分、默认波动、旧持仓或模拟数据补齐。

本功能不预测必涨基金，不承诺盈利，不代替用户确认销售平台状态，也不发送申购订单。

## 2. 决策链

```text
用户输入 2-6 只基金
  -> 可选填写本批次唯一总预算
  -> 确认资金尚未投入且不占用应急资金
  -> 每只基金独立执行真实数据 Agent Run
  -> 单基金确定性门禁输出适用性与容量，不输出金额
  -> 全部子 Run 到达终态
  -> 校验 Batch 输入哈希与全部 Run 结果哈希
  -> 校验真实年化波动和披露持仓覆盖
  -> 逆波动初始权重 + 已披露重合下界降权
  -> 应用逐只容量上限
  -> 联合检查权益与每个行业最坏上界
  -> 向下取整到分，残余资金保持未投入
  -> 固化不可变分配事件
  -> 用户人工复核，不自动下单
```

## 3. 创建批次时的资金契约

`POST /api/v1/agent/batches` 新增：

```json
{
  "codes": ["013403", "014089"],
  "planned_amount": 5000,
  "acknowledged_available_cash": true,
  "include_portfolio_context": true,
  "include_market_intelligence": true
}
```

约束：

| 条件 | 处理 |
| --- | --- |
| 不填写预算 | 允许纯研究；后续不生成金额 |
| 填写预算但未确认现金 | 请求拒绝 |
| 确认现金但没有预算 | 请求拒绝 |
| 预算小于等于零或超过 1 亿元 | 请求拒绝 |
| 填写预算但关闭组合上下文 | 请求拒绝 |
| 填写预算但关闭市场/披露持仓情报 | 请求拒绝 |

确认语义是“资金尚未投入、不包含已提交或待确认申购，并且不占用应急资金”。系统不读取银行或销售平台余额，因此该字段是用户确认事实，不是自动核验余额。

## 4. 子 Run 金额隔离

每个 Batch 子 Run 固定写入：

```json
{
  "allocation_scope": "portfolio_batch"
}
```

`personalized_fund_decision@1.4.0` 仍执行原有资料、市场权限、汇率风险、期限、回撤承受力、单品仓位、权益和行业门禁。若单基金具备新增条件，则动作改为 `batch_allocation_pending`，并满足：

- `budget.allowed_full_amount = null`；
- `budget.first_tranche_amount = null`；
- 保存 `pre_allocation_action`、最大候选容量和批次数；
- 保存当前持仓哈希、IPS 版本与哈希、穿透快照 ID 与哈希；
- 保存联合权益和行业计算需要的真实上界参数。

因此，即使六个子 Run 都通过单基金门禁，也不会产生六份完整预算。

## 5. 确定性分配方法

### 5.1 初始风险权重

对通过单基金门禁的候选 `i`：

```text
overlap_penalty_i = 1 / (1 + disclosed_overlap_sum_i / 100)
raw_weight_i = overlap_penalty_i / annualized_volatility_i
```

- `annualized_volatility` 只能来自该 Run 保存的真实基金分析事实。
- `disclosed_overlap_sum` 只累计本 Batch 已获取披露持仓的两两重合下界。
- 没有披露重合不代表完整组合不重合；页面必须继续显示该口径限制。
- 任一可分配候选缺少真实波动或披露持仓覆盖时，整次分配阻断。

### 5.2 容量约束

使用加权 water-filling 分配本批总预算。每只基金的分配不能超过单基金决策输出的确定性候选容量；达到容量的候选退出剩余资金分配，其余候选按权重继续分配。

### 5.3 联合组合约束

逐只门禁无法证明多只基金同时投入后仍满足组合上限，因此对整个候选向量统一求缩放系数。

对权益或任一行业约束：

```text
current_amount + scale * Σ(target_ratio_i * desired_amount_i)
  <= limit_ratio * (base_portfolio_total + scale * Σ desired_amount_i)
```

权益使用每只基金的权益上界。行业使用当前未知权益金额加该行业已知下界，并把候选基金未知行业比例叠加到每个行业的已知比例，形成保守最坏情形。最终缩放系数取权益和所有行业约束中的最小值。

若当前组合已超过上限，系统不会用新增投入假设自动修复超限；分配金额保持为零并提示先处理风险边界。

### 5.4 金额舍入

每只基金金额向下取整到 0.01 元。舍入残差、容量未用完部分和联合约束保留部分全部计入 `unallocated_total_yuan`，不重新强塞给其他候选。

## 6. 分配门禁

| 门禁 | 必须满足 |
| --- | --- |
| Batch 终态 | 所有子 Run 已完成、部分完成、失败、取消或明确 abstain |
| 唯一总预算 | Batch 输入中存在正数预算 |
| 可用现金确认 | 用户已做双向一致确认 |
| 组合上下文 | 批次启用了真实持仓和 IPS |
| 子 Run Evidence | 每只基金绑定持仓、IPS、穿透快照和目标代码 |
| 组合版本一致 | 全部 Run 的持仓哈希、IPS 版本/哈希和组合总额一致 |
| 披露重合覆盖 | 所有可分配候选都有真实披露持仓 |
| 联合暴露基座 | 权益和行业最坏上界参数完整且一致 |
| 可分配候选 | 至少一只基金通过单基金新增门禁 |

任一门禁失败，保存的快照状态为 `blocked`，分配金额为零。阻断快照同样不可修改；修复数据后应创建新 Batch，不能覆盖旧证据。

## 7. 不可变事件与并发一致性

新增 `agent_batch_allocation_events`，每个 Batch 最多一条事件：

- `batch_id` 唯一；
- 保存策略 ID、策略版本和 Schema 版本；
- 绑定 Batch 输入 SHA-256；
- 绑定按批次顺序排列的 Run ID、状态和结果 SHA-256；
- 保存载荷 SHA-256 和事件 SHA-256；
- 保存操作者与创建时间；
- `UPDATE` 和 `DELETE` 由 SQLite 触发器拒绝。

写入使用 `BEGIN IMMEDIATE`。事务内重新读取 Batch 和全部子 Run，并重新计算输入及 Run 集合哈希。研究结果在用户点击和事件写入之间发生变化时，写入失败并要求刷新。完全相同的重复请求返回原事件；不同载荷不能覆盖已存在事件。

## 8. API

```text
POST /api/v1/agent/batches/{batch_id}/allocation
```

请求：

```json
{
  "expected_batch_input_hash": "<64 位 SHA-256>"
}
```

响应返回事件 ID、是否首次创建以及包含分配快照的最新 Batch 摘要。输入哈希变化、子 Run 未终态或已存在不同快照时返回 `409`；批次未启用组合上下文时返回 `400`。

## 9. 大模型边界

提示模板升级为 `fund_decision_synthesis@2.1.0`。当确定性动作是 `batch_allocation_pending`：

- 模型状态必须是 `insufficient`，置信度必须是 `low`；
- 只能解释该基金为何进入组合资金复核；
- 不能输出投入金额、分配比例或下单动作；
- 不能把新闻、情绪或预测结果用于改变确定性金额；
- 最终金额只读取组合分配快照。

分配器明确记录 `return_forecast_used=false` 和 `model_amount_used=false`。

## 10. 前端交互

批量基金入口新增：

- 本批次唯一总预算输入；
- 尚未投入且不占用应急资金确认；
- 组合上下文和披露持仓情报依赖提示；
- 纯研究模式与可分配模式的明确区分。

批次完成后，用户显式点击“生成分配快照”。页面展示：

- 唯一总预算、已分配和保持未投入金额；
- 联合约束缩放比例和全部门禁；
- 每只基金的真实年化波动、已知披露重合累计、可验证容量；
- 组合分配金额、占已分配资金比例和首批人工复核金额；
- 阻断原因、口径警告和快照完整性状态。

该区域是 Batch 结果的一部分，不新增顶层功能入口，也不把组合分配包装成自动交易。

## 12. 当前限制

- 重合度仍是前 N 大已披露持仓下界，不是全量重合度，也没有统一不同基金的披露截止日。
- 当前分配以风险约束为主，不把未经样本外验证和发布审批的收益预测用于金额优化。
- SQLite 单实例事务适合当前部署；多实例前必须迁移 PostgreSQL，并让不可变事件和 Run 集合检查处于同一数据库事务。
- 快照只支持人工复核，不读取销售平台实时申购限额、费率或账户余额；真正交易前仍需独立的执行前报价和用户操作。

# 持久化自动结果观察调度

## 更新信息

- 日期：2026-07-13
- 更新编号：2026-07-13-002
- 能力版本：`outcome_schedule@1.0.0`
- 适用范围：具有可评分方向性动作的基金深度研究 Agent Run
- 状态：代码、本地集成测试和真实门禁验证完成，等待生产部署验收

## 决策：为什么必须做

上一更新已经能够把原决策与之后的真实确认净值绑定，但评估只能由用户手动点击。手动触发存在四个根本问题：

1. 用户不会在每个净值发布日逐只基金点击，结果样本会严重缺失。
2. 只在用户想查看时评估会形成选择偏差，无法用于策略版本治理。
3. 进程重启、数据源短暂故障和重复点击会让观察时间不连续。
4. 没有统一调度记录，就无法回答“为什么这个 Run 没有继续观察”。

因此，持久化自动观察不是便利功能，而是结果闭环从人工演示走向工业运行的必要基础设施。它不会直接创造收益，但能够持续积累淘汰无效策略所需的真实样本，是后续策略绩效注册表的前置条件。

## 被否决的简化方案

### 进程内定时器

仅用 Python 线程保存下次运行时间，服务重启后计划丢失，多进程部署时还会重复执行。因此只允许线程负责轮询，任务事实必须存入数据库。

### Systemd Timer 或 Cron 直接调用基金接口

系统定时器无法自然继承 Run 的 Evidence、用户范围、决策动作和审计链，也难以按 Run 暂停、退避和复核。它可以用于唤醒 Worker，但不能替代业务调度模型。

### 每天重新运行整份基金研究

重新研究会改变分析基线和策略输入，不能回答原建议之后发生了什么。本功能只运行版本化结果工具，不改写原 Run，不重跑原结论。

### 所有 Run 都自动观察

`setup_required`、`research_only` 等动作没有可评分方向。为它们请求净值只会消耗数据源配额，并可能诱导用户把普通市场涨跌误解成建议效果。因此只有方向性动作可进入自动计划。

## 资格门禁

一个 Run 必须同时满足以下条件：

- 意图为 `fund_deep_research`。
- 状态为具有结果的终态。
- 基金代码有效。
- 保存了不可变的确认净值日期和单位净值。
- 全部 Evidence 哈希和 Run 审计链通过校验。
- 原动作属于 `consider_tranche`、`wait`、`do_not_add`、`hold_no_add` 或 `reduce_exposure`。

资格判断输出 `eligible`、`reason` 和 `action`。非方向性 Run 返回 `decision_not_directional`，API 拒绝启用计划。

## 持久化数据模型

新增 `agent_outcome_schedules`：

| 字段 | 作用 |
|---|---|
| `id` / `run_id` | 计划标识；每个 Run 最多一条计划 |
| `tenant_id` / `user_id` | 保留未来租户与用户隔离边界 |
| `status` | `active` 或 `paused` |
| `interval_hours` | 12 至 168 小时 |
| `next_run_at` | 下一次可认领时间，统一使用 UTC |
| `lease_owner` / `lease_expires_at` | 跨进程执行租约 |
| `attempt_count` | 累计执行次数 |
| `consecutive_failures` | 连续失败次数，用于退避与熔断 |
| `last_started_at` / `last_finished_at` | 最近执行生命周期 |
| `last_success_at` | 最近成功检查时间 |
| `last_provider_as_of` | 最近真实数据源截止日 |
| `last_evidence_id` | 最近保存或复用的 Outcome Evidence |
| `last_error_code` / `last_error_message` | 最近失败原因 |

计划通过外键绑定原 Run 和 Evidence。数据库使用立即事务进行认领、配置和状态提交，避免并发 Worker 同时获得同一计划。

## Worker 与租约

```text
查询 active 且到期的计划
  -> BEGIN IMMEDIATE
  -> 校验原租约为空或已过期
  -> 写入 worker_id 和 120 秒租约
  -> 追加 outcome.schedule.started 审计事件
  -> 提交事务
  -> 统一 DecisionOutcomeService
  -> 真实确认净值工具
  -> 幂等追加 Outcome Evidence
  -> 持有租约的 Worker 提交成功或失败状态
```

租约规则：

- 同一时点只有一个 Worker 能认领计划。
- Worker 崩溃后，租约到期即可由其他 Worker 接管。
- 旧 Worker 在租约过期后返回时不能覆盖新 Worker 的状态。
- 用户修改运行中的计划频率不会释放在途租约。
- 用户暂停计划会释放租约；已返回的真实 Evidence 仍可保留，但旧 Worker 不能恢复计划。

当前执行工具最长 45 秒，租约为 120 秒，保留网络清理与数据库提交余量。

## 失败退避与熔断

错误分为可重试和不可重试：

- 数据源超时、连接失败等暂时错误：可重试。
- 原 Evidence 损坏、基线无效、Run 类型不支持等确定性错误：不可重试。

连续可重试失败的计划：

1. 第一次：15 分钟后重试。
2. 第二次：1 小时后重试。
3. 第三次：4 小时后重试。
4. 第四次：12 小时后重试。
5. 第五次：耗尽重试预算，自动暂停。

不可重试错误立即暂停。暂停状态、错误码、失败次数和下次运行时间写入数据库与审计链，不能静默失败。
用户确认外部问题已处理并手动恢复计划时，连续失败预算和当前错误会清零；累计执行次数及历史审计事件保留。

## 自动纳入与历史回填

- 新的方向性 Run 完成后，由 Agent Worker 终态回调幂等创建 24 小时计划。
- 用户手动暂停后，终态回调和服务重启不会擅自恢复。
- Worker 启动时查询既有、尚未建立计划且动作方向明确的 Run，最多回填 100 条。
- 回填查询在数据库中直接筛选方向性动作，避免大量非方向性历史任务阻塞候选。
- 多进程同时回填时，`run_id` 唯一约束和立即事务保证只创建一条计划。

## 统一评估服务

手动评估与自动调度现在共同使用 `DecisionOutcomeService`：

- 统一校验 Run、Evidence、审计链和确认净值基线。
- 统一调用 `fund.decision_outcome.get@1.0.0`。
- 统一执行 45 秒超时。
- 统一保存幂等 Outcome Evidence。
- 统一返回数据库实际持久化载荷及其 SHA-256。
- 统一错误码、重试属性和 HTTP 映射。

这消除了“页面手动评估正确、后台调度走另一套逻辑”的双实现风险。

## API

### 查询计划

`GET /api/v1/agent/runs/{run_id}/outcome-schedule`

返回：

- 资格状态与拒绝原因。
- 公开计划状态、频率、下次执行和最近结果。
- 不返回租户字段、用户字段、租约所有者或租约到期时间。

### 配置计划

`PUT /api/v1/agent/runs/{run_id}/outcome-schedule`

请求字段：

- `enabled`：启用或暂停。
- `interval_hours`：12 至 168。
- `run_immediately`：把下一执行时间设为当前时刻，交给 Worker 认领。

API 不直接在请求线程执行网络任务。“立即执行”也是持久化入队，避免页面超时和重复任务。

## 前端

Agent 的决策结果区域新增持久化自动观察控制：

- 展示是否具备自动观察资格及明确原因。
- 使用复选开关启用或暂停。
- 支持 24、48、72、168 小时频率。
- 支持立即排队。
- 展示计划状态、下次执行、最近成功、连续失败和最近错误。
- 活跃计划每 15 秒刷新状态和 Outcome Evidence。
- 明确说明自动观察不会修改原决策、推断用户成交或自动下单。

## 测试证据

新增和更新的自动化覆盖：

- 新方向性 Run 自动建立计划。
- 非方向性 Run 不建立计划。
- 用户暂停不会被启动回填恢复。
- 历史方向性 Run 幂等回填。
- 同时只有一个 Worker 获得租约。
- 租约过期后可接管。
- 旧 Worker 不能提交过期租约。
- 在途计划修改频率不释放租约。
- 可重试错误按 15 分钟、1 小时、4 小时、12 小时退避。
- 连续第五次失败熔断暂停。
- 熔断后手动恢复会重置连续失败预算但保留历史审计。
- 不可重试错误立即暂停。
- Worker 成功记录真实数据截止日和 Evidence ID。
- API 启停状态持久化。
- API 不泄露内部租约和租户字段。
- 原手动评估幂等、Evidence 完整性和路由契约继续通过。

## 生产验收

部署后将在本节记录：生产提交、服务状态、云端测试、Worker 启动日志、公网资格接口、真实历史回填数量和审计结果。

当前真实历史没有可评分方向性动作，因此生产验收不会伪造正向计划。正向租约执行由同版本云端隔离数据库集成测试证明；真实公网正向计划将在用户完成真实投资配置并产生方向性决策后自然建立。

## 风险与后续条件

当前 SQLite 租约适合现阶段单机和少量多进程部署。迁移 PostgreSQL 后，应使用 `SELECT ... FOR UPDATE SKIP LOCKED` 或工作流平台实现同一语义。当前计划仍运行在 API 进程内的独立 Worker 线程中，事实状态已与进程解耦，因此可以在下一阶段无损拆分为独立 Worker 服务。

本能力只保证稳定积累真实结果，不代表策略具备盈利能力。只有累计足够 Outcome Evidence，并按策略版本、市场、基金类别和观察周期分层后，才能建设策略绩效注册表并决定暂停或淘汰策略。

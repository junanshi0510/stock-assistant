# 2026-07-15 更新 009：多基金批量申购执行前复核

## 1. 解决的问题

组合分配回答的是“一笔总预算在候选基金之间最多如何分配”，但不能证明这些金额此刻可以在销售平台执行。实际申购还存在费用、暂停申购、单日限额、确认周期、跨境权限、汇率风险和组合事实变化。

本轮在组合分配之后增加一个独立的执行前复核阶段。它不连接交易账户、不读取余额、不发送订单，只把用户从销售平台本次申购页确认的事实与服务器重新获取的真实数据绑定起来。

## 2. 固定决策顺序

```text
不可变组合资金分配
  -> 逐只录入销售平台本次申购页事实
  -> 校验报价时间、费用、状态、限额和确认日
  -> 校验逐只金额与整批唯一预算
  -> 重新读取当前持仓和激活 IPS
  -> 重新读取基金市场画像
  -> 重新读取组合内全部基金定期披露
  -> 投影扣除申购费后的全部持仓
  -> 联合校验单品、权益和行业上限
  -> 固化不可变复核修订
  -> 仅进入人工申购确认
```

任何一步缺少真实事实都会阻断。系统不会用标准费率、历史优惠、默认限额、旧持仓或模拟披露替代。

## 3. API 契约

新增：

```text
POST /api/v1/agent/batches/{batch_id}/purchase-preflight
```

请求示例：

```json
{
  "expected_allocation_event_id": "batch_allocation_...",
  "expected_allocation_event_hash": "64位SHA-256",
  "expected_previous_event_hash": null,
  "acknowledged_platform_quotes": true,
  "quotes": [
    {
      "code": "013403",
      "platform_name": "销售平台全称",
      "quoted_at": "2026-07-15T10:30:00+08:00",
      "currency": "CNY",
      "order_amount_yuan": 2000.00,
      "entry_fee_yuan": 0.00,
      "purchase_status": "limited",
      "purchase_limit_yuan": 5000.00,
      "expected_confirmation_date": "2026-07-17"
    }
  ]
}
```

约束：

| 字段 | 规则 |
| --- | --- |
| 基金集合 | 必须精确覆盖所有分配金额大于 0 的基金，不允许缺失、额外或重复 |
| 报价时间 | 必须包含时区，不能晚于服务器超过 5 分钟，提交时不能超过 24 小时 |
| 申购金额 | 大于 0，不超过该基金分配金额；整批合计不超过已分配金额和原始总预算 |
| 实际申购费 | 可申购或限购时必须来自本次确认页，且大于等于 0、小于申购总额 |
| 限购 | 状态为 `limited` 时必须提供大于等于拟申购金额的真实限额 |
| 确认日期 | 可申购或限购时必须提供，范围为报价日起 30 天内 |
| 币种 | 当前只接受销售平台人民币扣款金额，不做客户端汇率换算 |

## 4. 组合联合门禁

`portfolio_batch_purchase_preflight@1.0.0` 固定检查：

1. 组合分配事件完整且仍是当前快照。
2. 当前持仓哈希和 IPS 版本/载荷哈希仍与分配阶段一致。
3. 每只基金的平台事实完整且报价在有效期内。
4. 全部申购金额只消耗本批次唯一预算。
5. 基金市场已由真实元数据识别，用户已开放对应市场并确认必要的汇率风险。
6. 扣除真实申购费后，拟申购基金的组合占比不超过 IPS 单品上限。
7. 当前持仓与全部拟申购金额均有可用、未过期且无冲突的真实基金定期披露。
8. 申购后的权益最坏上界和单行业最坏上界不超过 IPS。

输出状态：

- `ready_for_manual_purchase_review`：全部门禁通过，只可进入人工确认。
- `purchase_preflight_blocked`：提交时至少一道门禁失败。
- `expired`：保存后最早的平台报价超过 24 小时。
- `superseded`：持仓、IPS 或组合分配绑定发生变化。
- `integrity_failed`：内容哈希或事件链校验失败。

任何状态都固定包含：

```json
{
  "execution_authorized": false,
  "automatic_purchase_allowed": false,
  "order_submitted": false
}
```

## 5. 证据与审计

每次复核保存到 `agent_batch_purchase_preflight_events`，字段包括：

- 用户、租户和 Batch；
- 组合分配事件 ID、事件哈希和载荷哈希；
- 本次平台请求哈希；
- 持仓、投影持仓、IPS、市场画像、基金披露和组合穿透哈希；
- 策略与 Schema 版本；
- 前一修订哈希、当前事件哈希、操作者和服务端时间。

数据库触发器拒绝更新和删除。相同基础修订与相同输入的网络重试返回原事件；用户从最新事件发起重新复核时，必须带上最新前序哈希，防止两个页面同时覆盖彼此结果。

## 6. 前端交互

批量页保持固定层级：

1. 逐只基金研究结果。
2. 组合资金分配。
3. 申购执行前复核。
4. 披露持仓重合下界。

申购复核逐行展示拟申购额、费用、平台状态、限额、预计确认日、报价失效时间和逐只阻断原因。页面同时展示整批现金流、12 道确定性门禁、动态绑定状态、修订号和审计链结果。

表单不会预填平台名称、时间、费用、限额或确认日；只有组合分配金额可以作为拟申购金额初值。用户每次重新复核都必须重新确认平台事实。

## 7. 明确不做

- 不接入销售平台账户或模拟登录。
- 不获取、保存或推断账户余额。
- 不自动点击申购、赎回或转换。
- 不用基金标准费率替代本次平台费用。
- 不把“门禁通过”解释为未来会上涨或必然盈利。
- 不让大模型修改金额、忽略阻断项或授权订单。

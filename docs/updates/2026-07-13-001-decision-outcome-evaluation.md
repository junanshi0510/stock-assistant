## 更新信息

- 日期：2026-07-13

- 适用范围：基金深度研究 Agent Run
- 状态：代码与本地测试完成，等待生产部署验收

## 为什么需要这项能力

现有 Agent 能够读取真实基金数据、生成个性化决策、保存 Evidence，并校验审计链，但只能回答“当时为什么这样建议”。如果不记录建议之后真实发生了什么，系统无法区分以下情况：

1. 策略在长期样本中确实具有稳定性。
2. 某次建议只是短期运气。
3. 策略在特定市场或基金类型中持续失效。
4. 用户并未执行建议，但系统错误地把市场变化算作用户盈利。

结果评估闭环是工业级投资 Agent 的必要基础设施，不是可选展示功能。它为后续策略淘汰、版本比较、风险预算调整和合规复核提供可信样本，但自身不承诺盈利。

## 本次范围

本次只完成一个闭环：以原 Agent Run 保存的确认净值和决策动作为不可变基线，按需获取之后的真实确认净值，生成新的不可变 Outcome Evidence。

包含：

- 基金决策结果评估器及版本化工具协议。
- 原 Run 基线绑定，禁止使用重新计算的历史基线。
- 5、20、60、120 个后续确认净值样本的观察里程碑。
- 同一数据截止日幂等，新截止日追加 Evidence。
- Outcome Evidence 载荷哈希和追加式审计事件。
- 结果查询、重新评估 API 和 Agent 前端结果观察面板。
- 数据不足、非方向性建议和数据源失败时的明确拒绝或等待状态。

不包含：

- 不推断用户是否实际申购、赎回或调仓。
- 不把基金净值变化等同于用户账户收益。
- 不计算手续费、税费、申赎确认差异或用户实际汇率成本。
- 不在当前版本宣称超额收益，不与适配基准比较。
- 不自动下单，不根据单次结果自动修改策略。

## 核心规则

### 不可变基线

基线来自原 Run 保存的基金代码、确认净值日期、确认单位净值和个性化决策动作。旧 Run 缺少显式单位净值时，只允许从该 Run 已保存且通过完整性校验的 `latest_nav` Claim 中读取，禁止读取当前接口后再倒填。

### 时间隔离

评估器只接受日期严格晚于基准日的真实确认净值。基准日前和基准日当天的数据不会计入观察样本，从机制上防止时间穿越和回测污染。

### 最小观察样本

- 0 个后续确认净值：`pending`。
- 1 至 19 个后续确认净值：`observing`。
- 至少 20 个后续确认净值：`evaluable`。

少于 20 个样本时，系统不得宣称策略成功或失败。5 个样本只用于提示观察进度，不构成方向结论。

### 动作解释

- `consider_tranche`：20 个样本后只评价新增风险暴露的绝对净值方向。
- `wait`、`do_not_add`、`hold_no_add`：只描述暂缓新增暴露后的市场机会成本或下行回避，不称为用户盈利。
- `reduce_exposure`：只描述被降低部分的后续市场方向，不推断用户实际执行。
- `setup_required`、`research_only` 等非方向性动作：永久标记为 `not_scored`，不会根据后续涨跌反向包装成成功建议。

## 数据与证据链

```text
原基金研究 Run
  -> 校验全部原 Evidence 哈希
  -> 校验原审计链
  -> 读取保存的确认净值基线与动作
  -> fund.decision_outcome.get@1.0.0
  -> 获取真实历史确认净值
  -> 严格过滤基准日之后的数据
  -> 生成结果观察
  -> 追加 outcome_observation Evidence
  -> 追加 evidence.created 审计事件
  -> 追加 outcome.evaluation.created 审计事件
```

每份 Outcome Evidence 保存：

- 评估器 ID 和版本。
- 原 Run ID、完成时间、结果 Schema 版本和审计链头。
- 原决策动作及确认净值基线。
- 最新真实确认净值、数据截止日、样本数和绝对净值变化。
- 5、20、60、120 样本里程碑。
- 解释状态、方法约束、数据源和限制声明。
- 完整载荷 SHA-256。

## 幂等与并发

幂等键由 `run_id + evidence_type + provider_as_of + evaluator_version` 构成。数据库写入使用立即事务：

- 相同 Run、相同净值截止日、相同评估器版本只保存第一份 Evidence。
- 重复请求始终返回数据库已保存的载荷和对应哈希，不返回未持久化的临时重算结果。
- 数据源出现新确认净值日期时，追加一份新 Evidence，不覆盖历史记录。

这一规则允许复核“系统在每个数据时点知道什么”，同时避免重复点击污染样本。

## API 与工具协议

### Agent 工具

- 名称：`fund.decision_outcome.get`
- 版本：`1.0.0`
- 风险级别：R0，只读
- 超时：45 秒
- 输入：基金代码、基准日期、基准确认净值、原动作
- 输出：版本化结果观察、里程碑、解释、数据质量和来源

### HTTP API

- `POST /api/v1/agent/runs/{run_id}/evaluate`：基于真实确认净值建立或更新观察。
- `GET /api/v1/agent/runs/{run_id}/evaluations`：按数据截止日倒序读取不可变结果 Evidence。

接口拒绝以下请求：

- Run 不存在或尚未形成终态研究结果。
- Run 不是基金深度研究。
- 原 Run Evidence 或审计链校验失败。
- 缺少六位基金代码、确认净值日期或有效净值基线。
- 真实净值工具超时或数据源失败。

## 前端交互

Agent 结果页新增“决策结果观察”区域：

- 展示原决策基线和最新确认净值。
- 展示后续确认净值样本数和绝对净值变化。
- 展示当前可评估状态和解释，避免把观察结果写成盈利承诺。
- 展示 5、20、60、120 样本里程碑。
- 支持打开对应 Outcome Evidence 检查来源、载荷和哈希。
- 重复评估不会创建重复记录。

## 测试与验收

自动化覆盖：

- 基准日之前和当天的净值不进入观察窗口。
- 无后续净值时保持等待。
- 少于 20 个样本时不宣称成功或失败。
- 20 个样本后按原动作解释方向。
- 非方向性动作永久不评分。
- 相同截止日幂等，新日期追加 Evidence。
- 重复重算结果与已保存载荷不一致时，API 仍返回已保存 Evidence。
- Outcome Evidence 哈希完整性和追加审计链有效。
- API 路由契约保持稳定。

- 前端：`npm run build`，生产构建通过。

## 已知限制与下一步决策条件

本版本只能评价建议后的基金绝对净值路径，不能证明 Agent 帮用户赚到了钱。要评价真实盈利，必须先获得用户确认的交易流水、份额、费用、现金流和实际执行时间；要评价策略能力，还必须加入适配基准、滚动样本、市场分层和策略版本对照。

只有当本功能在生产环境稳定积累不可变结果样本后，才应进入下一步“策略版本绩效注册表”。在此之前直接展示胜率或自动调整策略，会制造统计幻觉。

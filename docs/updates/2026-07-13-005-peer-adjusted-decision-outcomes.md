# 2026-07-13 更新 005：同类基准校正的决策结果评估

## 1. 更新结论

本轮只交付一个能力：将基金 Agent 的决策结果评估从“只看后续单位净值涨跌”升级为“同时观察单位净值，并使用来源原生、同口径的基金累计收益与同类平均累计收益评价相对表现”。

- 结果评估器：`fund_decision_outcome@1.1.0`
- Agent 工具：`fund.decision_outcome.get@1.1.0`
- Evidence 类型：沿用不可变 `outcome_observation`
- 数据范围：公开基金，覆盖 A 股基金、债券基金、港股/美股等 QDII；是否可评取决于真实来源是否返回可精确对齐的同类序列
- 交易边界：只读，不下单，不推断用户是否执行

## 2. 为什么必须做

更新前的结果评估只能回答：“原决策之后，这只基金的单位净值涨了还是跌了？”它不能回答：“这只基金是否比同期可比基金更好？”

例如：

- 基金上涨 3%，同类平均上涨 8%。
- 旧评估会把新增暴露判断为“方向暂时有利”。
- 但这个结果不能证明选中了更好的基金；标的实际上落后同类约 5 个百分点。

如果继续用绝对涨跌给策略贴上“有利/不利”标签，后续策略注册、淘汰、Shadow 和 Canary 都会使用错误监督信号。系统可能奖励只是在牛市里随市场上涨的策略，也可能惩罚在弱市中明显抗跌的标的。

因此，相对结果不是展示增强，而是策略工业化和盈利能力评估的必要前置证据。它仍不能保证盈利，但可以避免把市场整体上涨误认成 Agent 的选基能力。

## 3. 为什么本轮不做其他功能

本轮没有增加新的预测模型、推荐榜单或聊天能力，原因如下：

1. 当前连历史建议的相对结果都不能正确评价，再增加预测只会扩大不可验证结论。
2. 策略发布门禁需要先获得口径一致的 Outcome Evidence，才能建立可信样本。
3. 用户真实成交、费用和现金流尚未完整绑定，当前不应声称计算了用户真实收益。
4. 公网身份与 HTTPS 仍是单独的生产安全前置项，但它需要域名和身份方案配合；本轮功能不扩大用户写入权限或敏感数据范围。

## 4. 被拒绝的快捷方案

### 4.1 按市场硬编码一个指数

没有采用“A 股一律沪深 300、港股一律恒生指数、美股一律标普 500”。债券、行业主题、成长风格、跨市场和混合资产基金与这些指数口径不一致，容易制造虚假超额。

### 4.2 使用页面数组第二项

没有假设 `Data_grandTotal[1]` 永远是同类平均。代码只接受来源明确命名为“同类平均”的序列，标签不符即不可用。

### 4.3 用基金单位净值直接减同类累计收益

没有混用两个收益口径。单位净值可能因分红发生机械下降，而累计收益序列可以包含同一来源的收益处理。相对计算只允许使用同一来源中的基金累计收益序列和同类平均累计收益序列。

### 4.4 用最近日期代替缺失日期

没有用最近交易日、前值填充或线性插值。原决策基线日和观察日必须同时存在于基金与同类序列；任一日期缺失，相对评价即为 `unavailable`。

### 4.5 上游失败时返回旧缓存或模拟同类

没有增加模拟数据或静默代理基准。真实确认净值仍可形成绝对结果，但 Evidence 质量降为 `partial`，并显示同类不可用原因。

## 5. 真实数据契约

### 5.1 确认净值

确认单位净值继续来自东方财富基金净值走势或天天基金历史净值接口。原 Agent Run 保存的以下字段不可变：

- 基金代码。
- 原决策确认净值日期。
- 原决策确认单位净值。
- 原决策动作。

后续评估只读取严格晚于基线日的确认净值，不使用盘中估值。

### 5.2 同口径累计收益序列

详情页 `Data_grandTotal` 必须同时满足：

1. 存在名称精确为“同类平均”的序列。
2. 页面存在可解析的正式基金名称 `fS_name`。
3. 某一累计收益序列名称与页面基金名称精确一致，或双方规范化后存在至少 6 个字符的唯一前缀关系。
4. 基金序列只能唯一匹配；零个或多个候选均拒绝。
5. 两条序列各至少有 2 个有效点。
6. 有效累计收益必须大于 -100%。
7. 重复日期按同一响应中的最后值去重，再按日期排序。

该规则已验证可识别真实的截断名称，例如：

```text
页面基金名：华夏恒生科技ETF发起式联接(QDII)C
累计收益序列名：华夏恒生科技ETF发起
```

代码不依赖数组位置完成匹配。

## 6. 计算方法

### 6.1 单位净值变化

单位净值变化仅作为原 Run 基线后的观察值：

```text
单位净值变化率 = 观察日单位净值 / 基线日单位净值 - 1
```

界面明确标注“未含分红”，该值不与同类累计收益直接相减。

### 6.2 同类区间收益

详情页序列提供的是从统一序列起点开始的累计收益。任意两个日期之间的收益不能直接用累计收益相减，必须还原复合收益：

```text
同类区间收益
= (1 + 同类观察日累计收益) / (1 + 同类基线日累计收益) - 1
```

### 6.3 基金同口径区间收益

```text
基金同口径区间收益
= (1 + 基金观察日累计收益) / (1 + 基金基线日累计收益) - 1
```

### 6.4 收益差与相对超额

```text
收益差（百分点） = 基金同口径区间收益 - 同类区间收益

相对超额收益
= (1 + 基金同口径区间收益) / (1 + 同类区间收益) - 1
```

界面同时显示收益差和复合相对超额，避免把百分点与收益率混为一谈。

## 7. 日期门禁

日期策略固定为 `exact_provider_date_only`：

- 基线日必须同时存在于基金累计收益和同类平均累计收益序列。
- 当前观察日必须同时存在于两条序列。
- 5、20、60、120 个确认净值里程碑分别执行同样门禁。
- 不向前找最近值，不向后使用未来值，不插值。

主要不可用原因：

| 原因 | 含义 |
|---|---|
| `provider_comparable_series_missing` | 基金或同类序列不存在 |
| `baseline_date_not_in_provider_comparable_series` | 原决策日不能同口径对齐 |
| `observed_date_not_in_provider_comparable_series` | 当前观察日不能同口径对齐 |
| `invalid_provider_comparable_cumulative_return` | 累计收益无法转换为正的复合净值 |
| `provider_peer_series_unavailable:*` | 来源抓取、标签或名称绑定失败 |

## 8. 结果解释规则

至少获得 20 个后续确认净值样本后才形成方向标签。相对序列可用时，解释使用基金同口径累计收益，不使用未含分红单位净值替代。

新增暴露类动作：

| 基金同口径收益 | 相对超额 | 结果标签 |
|---:|---:|---|
| 正 | 正 | 上涨且暂时跑赢同类 |
| 正 | 非正 | 上涨但暂时跑输同类 |
| 非正 | 正 | 下跌但暂时强于同类 |
| 非正 | 非正 | 下跌且暂时跑输同类 |

等待、不新增和降低暴露动作继续只描述“未新增/被降低部分”的市场后果，不推断用户真实执行或真实收益。

同类序列不可用时，系统保留旧的绝对方向观察，但明确写明“不评价相对表现”，不会生成选择优势结论。

## 9. Evidence 与审计

### 9.1 版本追加

- 旧 `fund_decision_outcome@1.0.0` Evidence 不修改。
- 新评估保存为 `fund_decision_outcome@1.1.0`。
- 同一 Run、同一数据截止日、同一评估器版本保持幂等。
- 同截止日的 1.0.0 与 1.1.0 可以并存，列表按创建时间显示新版本在前。

### 9.2 可复现字段

新 Outcome Evidence 保存：

- 原 Run 基线和动作。
- 后续确认净值及样本数。
- 基金和同类在基线日、观察日的累计收益值。
- 基金同口径区间收益、同类区间收益、收益差和相对超额。
- 公式版本、对齐策略、来源和来源日期。
- 来源序列起止日期与观察点数量。
- 不可用原因和质量状态。

原始整条滚动序列不写入用户结果，但计算所需的四个端点已进入不可变 Evidence，可重新计算本次结果。

### 9.3 质量状态

- 基金与同类序列可用，或已完成基线对齐但尚无后续净值：`complete`。
- 绝对确认净值可用，但同口径相对序列不可用：`partial`。
- 未知质量值按 `partial` 保存，不能默认为完整。

Evidence 哈希和 Agent 审计链继续使用现有完整性校验。

## 10. 自动观察调度

持久化 Outcome Worker 自动使用工具 1.1.0：

- 调度计划、租约、退避和失败预算不变。
- 同类来源失败不会丢弃已经取得的真实确认净值；本次执行以 `partial` Evidence 成功结束，并保留不可用原因。
- 工具超时后仍不允许迟到线程写 Evidence；持久化只发生在主流程确认工具完成之后。
- 自动观察不修改原决策、不推断成交、不下单。

## 11. 前端变化

Agent 结果页的 Outcome 区域新增：

- “单位净值变化”，明确标注未含分红。
- 来源原生同类基准名称与精确日期对齐状态。
- 同类区间收益。
- 基金同口径区间收益。
- 收益差（百分点）。
- 复合相对超额。
- 每个已达到里程碑的同类收益和相对收益。
- 同类平均不是合同业绩基准、单次结果不能证明持续超额的边界说明。

来源缺失时显示原因，不隐藏相对评价不可用状态。

## 12. 测试覆盖

本轮新增或强化测试：

- 只接受明确的“同类平均”标签。

## 15. 安全与隐私

- 新工具为 R0，只读取公开基金数据。
- 不新增用户写入 API。
- 不上传文件，不向第三方发送用户持仓。
- Outcome 只绑定服务端从原 Run 读取的代码、日期、净值和动作，客户端不能替换基线。
- 原 Run Evidence 或审计链校验失败时拒绝评估。
- 当前公网 HTTP 和无认证问题仍存在；正式多用户开放前必须完成域名、HTTPS、身份、授权和数据库行级隔离。

## 16. 已知限制

1. “同类平均”是东方财富页面的同类比较序列，不等于基金合同业绩比较基准。
2. 页面当前提供滚动累计收益窗口；过旧基线滚出窗口后，相对评价会变为不可用，不用其他指数补齐。
3. 单次基金结果不能证明策略具有统计显著的持续超额。
4. 当前不包含用户成交时间、份额、申赎费、税费、现金分红到账和汇率换汇成本。
5. 单位净值变化仍不含分红，只作为原不可变基线的辅助观察；相对评价使用来源同口径累计收益。
6. 来源授权和商业使用范围仍需在公开商业化前完成数据授权台账。

## 17. 回滚方案

如工具 1.1.0 出现生产问题：

1. 回滚应用到上一稳定 Git 提交。
2. Worker 将恢复注册工具 1.0.0。
3. 已保存的 1.1.0 Evidence 保持不可变，不删除、不降级、不覆盖。
4. 前端旧版本会继续显示单位净值结果，不读取新增同类字段。
5. 恢复后可通过新版本重新评估，不修改历史。

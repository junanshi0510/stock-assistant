# 持仓行动中心与不可变组合报告

## 1. 更新信息

- 更新编号：`2026-07-13-009`
- 功能：真实持仓行动中心、逐行持仓详情、基金重复暴露
- 报告协议：`portfolio_action_report.v1`
- 规则版本：`portfolio_action_rules.v1`
- 数据范围：用户确认持仓、已激活投资政策、用户交易流水、真实基金净值、基金定期报告
- 存储：SQLite 不可变行动报告
- 完整性：持仓事实哈希、投资政策版本绑定、报告载荷 SHA-256、数据库更新触发器
- 性能边界：基金趋势与基金重合度共用 30 秒整体截止时间
- 决策边界：只排序风险控制和证据补全动作，不承诺盈利，不自动交易

## 2. 为什么本轮必须做

原组合页已经存在持仓汇总、基金趋势、基金重合度、组合穿透、交易账本、现金流收益和再平衡，但这些能力被拆散在不同页面和区块中。用户需要自己完成以下推理：

1. 哪个数据缺口会阻止投资判断。
2. 哪只持仓违反本人设置的风险上限。
3. 多只基金是否只是重复持有相同股票或行业。
4. 当前亏损是否足以支持补仓、持有或卖出。
5. 某条建议使用了哪一天、哪份披露和哪个投资政策版本。

这会产生两个实际风险：

- 页面信息很多，但没有明确的下一步操作顺序。
- 旧持仓分析可能在持仓金额或投资政策变化后继续被误用。

本轮把已有真实数据收敛为一个版本化、可审计、默认拒绝的持仓行动协议。目标不是预测必然上涨，而是先减少无意集中、重复暴露、无效换手和错误收益口径对长期复利的破坏。

## 3. 被否决的方案

### 3.1 根据浮亏比例自动补仓

浮亏率不包含未来收益信息。下跌可能来自估值回归、资产质量变化、市场风格变化、汇率或原持有逻辑失效。报告只要求复核亏损来源，不把浮亏直接转换成加仓信号。

### 3.2 根据短期上涨自动追入

短期动量不能绕过用户风险边界、基金市场权限、重复暴露和策略发布门禁。本轮不生成追涨指令。

### 3.3 用基金名称判断重复度

名称相似不等于持仓相同，名称不同也可能持有相同股票。本轮只使用基金定期报告披露的共同持股和行业占比。

### 3.4 数据源失败时沿用旧结论

旧报告可能对应不同金额、收益、份额或投资政策。本轮对全部决策相关持仓字段生成哈希；绑定不一致时前端停止展示旧操作状态。

### 3.5 为了速度减少到两只或三只基金

固定截断会隐藏组合后部的重复暴露。本轮支持按金额排序的最多 8 只基金，并通过受控并发解决串行等待问题。超过范围时不会声称分析了全部持仓。

### 3.6 每只基金独立给买卖结论

单品分析无法识别组合集中、行业叠加、权益总仓位和共同持股。行动顺序必须先看组合，再看单品。

## 4. 报告输入

### 4.1 用户确认持仓

进入持仓哈希的字段：

- 资产类型、市场、代码、名称。
- 当前金额、成本、昨日收益、累计收益、收益率、份额。
- 数据来源和更新时间。

任一字段变化都会使旧报告失效。

### 4.2 已激活投资政策

报告只读取已激活版本：

- 单品仓位上限。
- 权益总仓位上限。
- 单行业上限。
- 最大可承受回撤。
- 投资政策版本 ID。

未激活时，默认值不会进入操作计算，全部持仓保持 `data_required`。

### 4.3 交易与收益

- FIFO 成本与剩余仓位来自用户交易流水。
- 资金加权收益只在流水覆盖完整时可用。
- 缺少流水时要求补录期初持仓，不猜测历史买卖。

### 4.4 真实基金数据

- 趋势、区间收益和回撤来自真实基金净值。
- 重复暴露来自基金定期报告共同持股和行业披露。
- 每只基金展示净值日期和披露期。
- 来源失败时返回具体错误，不使用模拟数据或替代值。

## 5. 操作状态协议

操作优先级固定为：

```text
数据完整性
  > 已激活投资政策
  > 单品仓位超限
  > 中高重复暴露
  > 风险承受不匹配
  > 回撤与持有逻辑复核
  > 保持现有纪律
```

### 5.1 `data_required`

触发条件：

- 持仓金额缺失或无效。
- 投资政策未激活。
- 基金真实净值或风险证据未返回。
- 当前报告与持仓或投资政策版本不一致。

行为：停止对应投资操作，不生成新增或减仓金额。

### 5.2 `reduce_review`

唯一金额型减仓复核：当前持仓超过用户已激活的单品上限。

报告给出相对上限的超出金额，但它不是自动赎回指令。执行前必须核对赎回费、税费、流动性和持有逻辑。

### 5.3 `pause_add`

基金定期报告显示与组合内其他基金存在中高个股或行业重合时，暂停同时新增，先比较费用、跟踪目标、基金经理稳定性和长期同类表现。

### 5.4 `risk_review`

基金历史最大回撤超过用户确认的最大可承受回撤时，在完成风险适配复核前不扩大仓位。

### 5.5 `thesis_review`

当前回撤达到规则阈值时，保持仓位并复核原持有逻辑。回撤本身不触发卖出或补仓。

### 5.6 `hold_review`

未触发上述门禁时保持现有仓位并按计划复核。它不等于看涨，也不自动放行新增金额。

## 6. 提高盈利概率的策略顺序

报告生成组合级步骤，而不是把每只基金的短期走势拼成交易清单：

1. 补齐金额、份额和交易流水，先得到可核对的费用后收益。
2. 激活个人投资政策，让风险边界来自用户确认值。
3. 处理超过单品上限的无意集中。
4. 暂停向中高重合基金同时新增，先明确每只产品的组合角色。
5. 对主要亏损来源复核市场、重复暴露、风险适配和原持有逻辑。
6. 没有触发问题时减少无效换手，不因单日涨跌交易。

这套顺序不能保证正收益。它的可验证价值是减少集中损失、重复风险、追涨杀跌和错误绩效口径。

## 7. 重复度算法

对每对基金分别计算：

```text
个股重合 = Σ min(基金 A 中共同股票占比, 基金 B 中共同股票占比)
行业重合 = Σ min(基金 A 中共同行业占比, 基金 B 中共同行业占比)
```

等级沿用既有真实披露规则：

- 高度重合。
- 中度重合。
- 低度重合。
- 重合较低。

报告保存：

- 每对基金的个股和行业重合率。
- 共同持股数量和主要共同持股。
- 主要共同产业。
- 两只基金各自披露期。
- 失败基金和失败原因。

定期报告通常滞后于实时净值，因此结果用于识别结构性重复风险，不声称代表实时持仓。

## 8. 不可变存储与绑定

新增 `portfolio_action_reports`：

| 字段 | 内容 |
|---|---|
| 身份 | report ID、user ID |
| 协议 | schema version、ruleset version |
| 输入绑定 | holdings SHA-256、profile version ID |
| 输出 | status、payload JSON、payload SHA-256 |
| 时间 | created_at |

数据库触发器拒绝任何 `UPDATE`。报告只追加，不原地修改。

读取报告时重新计算：

1. 报告载荷 SHA-256。
2. 当前持仓事实 SHA-256。
3. 当前已激活投资政策版本 ID。

任一不一致都会返回 `binding.current=false`。前端不展示旧报告中的操作步骤或逐持仓操作状态。

## 10. 性能与失败策略

原组合体检依次请求每只基金净值，完成后再请求基金重合度。三只基金真实联调超过 90 秒。

本轮改为：

- 基金趋势任务并发执行。
- 基金重合度与趋势任务并发执行。
- 最大并发数为 5。
- 全部基金数据共用 30 秒整体截止时间。
- 已成功真实结果保留，超时来源单独进入错误列表。
- 线程池不阻塞 HTTP 响应等待超时任务收尾。

本地真实联调中，3 只基金趋势成功返回 `3/3`；基金重合度源未在 30 秒内返回，被标记为 `unavailable`，没有生成替代重合结果。

## 11. 前端工作流

主页面收敛为三段：

1. 按优先级排列的下一步操作。
2. 每只持仓一行的持仓清单。
3. 基金成对重复暴露。

点击持仓行打开详情抽屉，展示：

- 当前操作状态及原因。
- 确认金额、占比、累计收益和份额。
- 真实净值日期、趋势、区间收益和回撤。
- 与其他基金的重复暴露和共同持股。
- 规则证据、数据来源和报告规则版本。
- 删除持仓操作。

维护持仓默认折叠；截图、CSV、Excel、粘贴文本和手动录入能力保留在维护区。保存持仓后旧报告立即失效。

移动端验证：

- 逐只持仓保持一行一项的结构化列表。
- 详情抽屉切换为全屏。
- 页面和抽屉均无横向溢出。

## 13. 当前限制

- 当前重合度依赖基金定期报告，不能代表实时基金持仓。
- 股票持仓暂未接入与基金相同的逐项策略发布门禁，缺少真实证券策略证据时只做持仓事实复核。
- 未完整录入交易流水前，系统不能证明策略提高了费用后真实收益。
- 行动报告最多分析 8 只基金的成对重合；超过范围需要后续引入批处理和持久化披露缓存。
- 当前产品仍是单用户迁移存储；多用户认证和租户隔离未在本轮实现。
- 报告不执行申购、赎回或自动交易。

## 14. 下一轮建议

下一轮不应继续增加更多提示语，应完成“持有逻辑台账”：

1. 用户为每只持仓保存买入理由、目标角色、复核日期和退出条件。
2. 行动报告把当前真实证据与原持有逻辑逐条比较。
3. 退出条件只能由用户确认或已发布策略触发，不能由模型临时改写。
4. 后续真实结果按报告版本追踪，检验每类操作是否改善用户的费用后收益和最大回撤。

# 金融投资助手

一个面向个人投资者的真实数据决策工作台，覆盖公募基金、A 股、港股、美股和用户真实持仓。

当前版本使用 **React + Vite** 构建前端，使用 **FastAPI + Python** 完成数据获取、基金与股票研究、组合账本、确定性计算和证据约束的大模型合成。项目正在按照工业级投资 Agent PRD 逐步升级；当前已经具备服务端登录、管理员/用户 RBAC、个人数据隔离、可恢复 Run、真实工具、Evidence、风险门禁和可选 LLM 研判，但仍不具备自动交易能力，也不把模型文本当作确定性计算结果。

> 风险提示：系统输出用于研究和风险复盘，不代表未来涨跌，不构成投资建议，也不承诺收益。数据源不可用、数据过期或用户持仓不完整时，系统必须明确显示缺口，不使用模拟数据补齐。

## 最近更新

### 2026-07-15：销售平台真实报价与换仓成本审计

- 披露成本核算结果现在按用户、持仓和候选基金保存为不可变内容寻址快照，并同时绑定持仓 ID、Evidence SHA-256 和完整载荷 SHA-256；重复读取只复用完全相同的证据，不跨持仓串账。
- 持仓详情可逐候选录入销售平台本次交易确认页显示的赎回费、候选申购费、报价时间、预计到账日和当前可申购状态。输入默认留空，不把费率页优惠快照冒充平台报价，也不把提交前报价冒充最终清算费用。
- 平台报价与披露成本区间进行确定性比较；偏差超过 `max(10 元, 赎回总额的 0.2%)` 时必须显式确认。费用超过赎回总额、报价超过 24 小时、核算日不一致或到账日异常时拒绝通过门禁。
- 每次更新报价都会追加新 revision，并通过前序哈希形成用户级不可变审计链；旧报价不会被覆盖。报价过期、候选不可申购、持仓/份额/FIFO 流水变化、当前成本快照被刷新或任一成本/审计哈希失败时，`executable_switch_cost_confirmed` 自动关闭。
- Agent 私有组合上下文只接收最新报价的候选代码、成本合计、成本率、在途天数、历史成本覆盖期和完整性状态，不发送平台名称或原始 FIFO 批次。只有报价当前有效、审计完整且成本门禁通过时，大模型才可称成本已确认；仍固定禁止自动交易。

### 2026-07-15：用户级 FIFO 换仓成本核验

- 新增 `fund_switch_cost_review@1.0.0`，把候选尽调结果与当前用户交易账本中的 FIFO 剩余买入批次绑定；不使用持仓创建时间、期初持仓日期或平均持有期代替真实申购确认日期。
- 基金费率解析新增可验证的金额上下界和持有天数上下界。每个剩余批次按核算日自然日数独立匹配当前基金赎回费档位，再按赎回后金额匹配候选基金申购费档位。
- 同时展示基金标准披露费率与费率页平台优惠快照，并用滚动 12 个月历史中位超额计算一次性成本的历史覆盖期；该数值是成本门槛，不是未来回本预测。
- 份额对账误差超过 0.1%、存在期初持仓批次、交易完整性失败、确认净值超过 7 天、费率区间无法唯一匹配或上游候选门禁未通过时，成本核算立即停止。
- 持仓详情按候选逐行展示 FIFO 日期、持有天数、命中费率、赎回费和两套换仓成本。即使披露成本完整，销售平台当日报价、到账时间和在途风险未确认前仍标记为不可执行，并固定禁止自动换仓。

### 2026-07-15：替代候选真实费率与披露持仓门禁

- 新增 `fund_alternative_due_diligence@1.0.0`，作为持续胜率后的第二阶段门禁；只有持续性已通过的候选，才有资格继续判断替换价值。
- 逐只读取天天基金费率档案中明确披露的管理费、托管费、销售服务费、金额分档申购费和持有期赎回费。年度运作费明确标记为已从净值扣除，不对历史收益重复扣费。
- 逐只读取基金最新可得定期报告持股和行业配置，按双方共同披露占净值比例的较小值计算重合下界。股票披露可用时以共同持股为主、宽泛行业分类为辅助，避免仅因都属于“制造业”就误判为同一组合。
- 候选被区分为差异化候选、部分重合、高重合但费率更低、高重合且无成本优势、费率缺口和披露不足；页面同步展示持股重合下界、行业重合、年度明确费率差和共同披露持股。
- 即使第二阶段通过，也只允许进入“用户逐笔持有天数 + 销售平台当日赎回报价”核验。没有真实持有批次时不选择赎回费率，继续固定禁止自动换仓、买入和赎回。

### 2026-07-15：替代候选持续胜率与追涨门禁

- 现有“替代审查”新增 `fund_alternative_durability@1.0.0`：使用数据源披露的每日收益复利形成总回报指数，只在当前基金与候选基金完全相同的月末日期上计算滚动 6/12 个月相对表现；不使用单位净值涨幅替代分红调整后的收益。
- 候选先按基金名称归并 A/C 等同策略份额，避免同一策略重复占据榜单并减少无效外部请求；页面保留被排除代码及原因供 Evidence 审计。
- 每只候选展示滚动胜率、中位超额、当前基金亏损窗口中的保护率、最大回撤差和当前 6 个月收益历史分位。近期领先但长期胜率不足标记为“仅近期领先”，持续占优但处于历史极热区时标记为“追涨区”。
- 只有滚动 6/12 个月胜率均不低于 60%、中位超额均为正、最大回撤不显著恶化且当前不处于极热区，才允许继续费用与持仓重合尽调；仍固定禁止自动买入和自动赎回。
- Agent 的同类候选 Evidence 同步加入持续性门禁，DeepSeek 不再只看到单年收益和初筛评分；原始逐日序列不发送给模型。

### 2026-07-15：持仓基金同类持续性诊断

- 单只基金持仓详情新增按需同类诊断，使用东方财富详情页中明确标记的基金累计收益序列和“同类平均”序列，按共同日期计算近 3/6 个月及两个非重叠季度；近 1 年读取同源真实阶段涨幅，并且只有在 3/6 个月基金与同类结果都通过 0.08 个百分点交叉校验后才接纳，不使用市场指数、单位净值或其他基金分类做代理。
- 新增 `fund_peer_relative_persistence@1.0.0`：分别检查最近和此前两个互不重叠的三个月窗口。只有两个窗口均跑输且近 12 个月落后至少 3 个百分点，才标记为“进入替代审查”，该阈值只是版本化研究门禁，不是卖出模型。
- 替代候选必须由用户再次点击后读取真实同类榜单和净值指标，不阻塞持仓列表。页面同时保留费用、份额类别、组合重合、基金经理和投资合同尚未核验的缺口，禁止自动赎回。
- Agent 新增 `fund.peer_persistence.get@1.0.0` R0 工具和独立 Evidence；DeepSeek 可以解释相对表现，但该证据不进入确定性仓位金额门禁。新 Run 使用 `fund_deep_research.v6`，历史 v4/v5 仍兼容策略跟踪。

### 2026-07-15：持仓基金同状态历史表现

- 基金持仓详情在“真实市场证据”中新增 3/6/12 个月历史前瞻分布，展示与最新确认净值具有相同 MA60 趋势和回撤区间的历史月末样本，而不是把盘中估值当作确认净值。
- 复用逐项估值回溯已经读取的最长 120 个月真实确认净值，不增加第二次历史抓取；批量结果继续按持仓隔离失败，并统计已评估和样本不足数量。
- 每个窗口展示正收益占比、中位收益、中间 50% 区间、样本日期和相对基金自身全历史的差异。少于 6 个同状态样本时只显示样本不足，不生成方向结论。
- `fund_conditioned_forward_return@1.0.0` 当前仍是影子研究策略：结果不进入持仓金额计算，不自动产生加仓或卖出动作，并明确披露前瞻窗口重叠、基金经理或合同变化未标准化等限制。

### 2026-07-15：逐项持仓当前估值历史到达

- “我的资产 → 持仓与纪律”在每一只基金或股票的行内显示当前估值/实时价位上一次历史到达结果，不再要求逐只进入基金或股票研究页查看。
- 新增用户级批量接口 `GET /api/holdings/level-recurrence`，按登录会话读取全部已确认持仓，使用最多 6 路受控并发；基础持仓列表先渲染，外部行情不会阻塞金额和收益展示。
- 股票继续使用腾讯证券实时成交价和真实未复权日线高低区间；基金继续使用东方财富盘中估算净值和历史确认净值。基金只能定位到相邻确认净值之间时显示“区间穿越”，并同时标明起止确认净值、被覆盖的当前估值和日期区间，不伪造精确时刻。
- 精确到达、区间穿越、覆盖期未到达和数据不可用分开呈现。单只来源失败不会拖掉其他持仓，也不会用昨收、确认净值或模拟数据兜底。
- 点击持仓行可查看目标值、数据时点、实时与历史来源、覆盖日期、样本数及口径限制；保存或删除持仓后自动重新读取批量结果。

### 2026-07-15：定时持仓检查与待办提醒

- 用户可在“今日决策”开启每日、每 3 日或每周真实数据检查，也可立即排队；计划默认关闭，不会在未经用户选择时批量抓取行情。
- 后台使用单并发租约 Worker，防止同一计划重复执行；程序异常按 15 分钟、1 小时、4 小时和 12 小时退避，连续失败耗尽后自动暂停。
- 检查结果区分完整完成、部分完成和执行失败。真实来源部分不可用时只新增可用性提示并保留旧风险，不会把缺失证据解释为风险消失。
- 检查配置、开始、完成、部分完成和失败事件进入用户级不可变 SHA-256 审计链；服务端不向前端暴露 Worker ID、租约或内部用户标识。
- 顶部“今日决策”新增待处理数量徽标，使用轻量摘要接口每分钟更新，不重复触发市场、基金或持仓分析。

### 2026-07-14：普通用户自助注册

- 未登录访问时只呈现登录/注册入口，不渲染持仓、行情、研究、Agent 或系统管理界面；所有业务 API 仍由服务端统一返回 `401`，前端隐藏不作为权限边界。
- 用户可只用账号和密码自行注册，无需管理员预先创建。注册接口固定创建 `user` 角色，不接受客户端传入角色或显示名，也不会自动登录或获得管理员权限。
- 注册密码继续使用 Argon2id 保存，账号唯一；注册请求按客户端哈希限流，并写入不可变认证审计链。管理员角色只能通过已认证的管理员接口授予。

### 2026-07-13：用户登录、管理员权限与个人数据隔离

- 新增服务端账户、角色、会话和权限系统。密码使用 Argon2id 保存；浏览器只接收随机会话 Cookie，Cookie 设置 `HttpOnly` 和 `SameSite=Lax`，数据库只保存 Token 哈希。
- 所有 `/api` 业务接口在生产环境统一经过登录、首次改密和 CSRF 门禁。匿名、过期、停用及角色变更后的会话会被服务端拒绝，前端不会把 Token 写入 `localStorage`。
- 管理员控制台支持创建账户、分配管理员/用户角色、启停账户、重置临时密码、查看全局用户/会话/Agent 统计和校验认证审计链；禁止管理员停用自己，也禁止系统失去最后一个启用管理员。
- 自选、提醒、持仓、投资政策、交易流水、组合快照、持有逻辑、行动报告、穿透快照和 Agent Batch/Run 全部绑定服务端会话中的 `subject_id`。普通用户访问他人 Run 返回 404；管理员可审计和处置跨用户 Run，但无法读取密码、会话 Token 或第三方 API Key。
- 旧版单用户数据一次性归属初始管理员；迁移有版本标记，不会在重启后把已经删除的旧自选或提醒重新写回。认证事件使用追加式 SHA-256 哈希链，数据库触发器禁止修改或删除。
- 新增离线 `bootstrap-admin`、`recover-admin` 和 `verify-audit` 运维命令；生产 systemd 模板改为独立低权限 Linux 用户和独立状态目录。

### 2026-07-13：DeepSeek 模型接入

## 数据原则

1. 不展示伪造行情、净值、财务数据、持仓或投资结论。
2. 可以在已接入的真实来源之间切换，但必须保留实际来源、数据时间和失败原因。
3. 来源全部失败时返回 `partial`、`unavailable` 或明确错误，不用示例值兜底。
4. 用户组合分析只使用用户保存或确认过的持仓和交易。
5. 盘中估值、确认净值、日终行情和定期报告披露不得混为同一种“最新数据”。
6. 收益率、回撤、FIFO、XIRR、相关性和组合集中度由确定性代码计算，不由大模型自行编造。
7. 港股、美股、全球和 QDII 基金必须披露市场识别依据、净值时差与汇率风险；无法确认具体市场时停止金额决策。
8. 只有相关真实来源完整返回时才能自动解决旧风险；部分失败必须保留旧任务并显式记录证据缺口。

## 快速开始

### 环境要求

- Python 3.10+
- Node.js 18+
- Windows 本地运行，或 Ubuntu 22.04/24.04 服务器

### Windows 一键运行

首次使用：

```text
双击 setup.bat
```

以后启动：

```text
双击 start.bat
```

浏览器会打开 [http://localhost:5173](http://localhost:5173)。后端 API 默认运行在 `http://127.0.0.1:8000`。

### 手动启动

后端终端：

```powershell
Set-Location C:\Project
.\venv\Scripts\Activate.ps1
Set-Location backend
python -m uvicorn main:app --reload --port 8000
```

前端终端：

```powershell
Set-Location C:\Project\frontend
npm install
npm run dev
```

开发环境中，Vite 会把 `/api` 请求代理到后端 `8000` 端口。FastAPI 接口文档位于 [http://127.0.0.1:8000/docs](http://127.0.0.1:8000/docs)。

本地开发默认保持 `AUTH_REQUIRED=false`，以兼容已有的一键启动流程；这只适用于本机开发。生产 systemd 模板固定启用认证，并且在审计 Pepper 或初始管理员缺失时返回 `503`，不会匿名放行。初始化生产管理员的完整步骤见 [DEPLOY.md](DEPLOY.md)。

| `AGENT_MAX_PENDING_RUNS` | Agent 排队和运行任务总上限；默认 `20` |

| `LLM_MODEL` | 明确批准的模型 ID，不提供隐式默认值 |
| `LLM_API_KEY` | 通用模型密钥；内置供应商优先使用下面的专用变量 |
| `OPENAI_API_KEY` | `LLM_PROVIDER=openai` 时的 OpenAI API Key |
| `DASHSCOPE_API_KEY` | `LLM_PROVIDER=dashscope` 时的阿里云百炼 API Key |
| `DEEPSEEK_API_KEY` | `LLM_PROVIDER=deepseek` 时的 DeepSeek API Key |
